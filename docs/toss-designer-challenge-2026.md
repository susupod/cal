# Toss Product Designer Challenge 2026 정리

출처: https://toss.im/career/designer-challenge-2026  
정리일: 2026-07-10

## 과제 한 줄 요약

**6명이 회의 일정을 잡는 경험을 설계한다.**

여러 사람의 일정, 선호, 필수/선택 참석 조건을 이해하고 모두에게 괜찮은 **1시간 회의 시간**을 찾도록 돕는 솔루션을 제안해야 한다.

## 문제 상황

같은 회사 동료 6명이 다음 주까지 모여야 한다.

조건:

- 회의 시간은 **1시간**
- 참여자는 총 **6명**
- 사람마다 일정과 선호가 다름
  - 누군가는 점심 직후를 피하고 싶어함
  - 어떤 사람은 특정 요일에 외근이 많음
- 참석 중요도가 다름
  - 꼭 참석해야 하는 사람
  - 선택 참석이어도 괜찮은 사람
- 기존 calendar app / scheduling service에서도 여러 조건을 한 번에 이해하고 모두에게 괜찮은 시간을 찾기 어려움

## 설계해야 하는 것

사용자가 다음을 할 수 있는 흐름:

1. 동료들의 상황을 파악한다.
2. 모두에게 괜찮은 후보 시간을 찾는다.
3. 회의 시간을 확정한다.

화면 형태와 환경은 자유:

- Mobile
- PC
- Tablet
- Web
- App
- 기타 어떤 형태든 가능

단, 핵심은 **사용자가 실제로 제품을 쓰는 흐름**이 보여야 한다.

## 제출 방식

제출 페이지에서 아래 3가지 질문에 답변한다.

1. **사용자의 어떤 문제를 발견했나요? 그것이 왜 문제라고 생각했나요?**
2. **그 문제를 어떻게 해결했나요?**
3. **왜 이 방식으로 설계했나요?**

결과물은 URL로 제출한다.

가능한 제출물:

- Figma prototype
- Website
- GitHub
- Video link
- 기타 사용 흐름을 볼 수 있는 URL

불가:

- Google Drive link

## 과제 범위와 주의사항

- 토스와 무관한 가상 과제
- 토스 앱과 연결해서 생각하지 않아도 됨
- AI 도구 사용 가능
  - Research
  - Prototype
  - UI design
  - 기타 전 과정
- 새 app을 설계해도 전체 branding / visual concept은 평가하지 않음
- 핵심 flow에 집중해야 함
- 문제 정의 없이 결과 화면만 제안하면 과제 취지에 맞지 않음
- 기존 화면을 그대로 따라 그리거나 visual style만 개선하는 것은 부적절

## 평가 기준

### 1. 문제 정의

주어진 단서와 맥락을 바탕으로 사용자가 겪는 **본질적인 문제**를 발견하고 정의했는지 본다.

우리 작업에서 보여줘야 할 것:

- “일정 잡기” 자체가 아니라 어떤 순간이 진짜 어려운지 정의
- 6명, 필수 참석자, 선택 참석자, 선호 시간, 외근, 점심 직후 회피 같은 조건이 왜 복잡도를 만드는지 설명
- 사용자가 현재 어떤 방식으로 판단하고, 어디서 헷갈리고, 무엇 때문에 시간이 낭비되는지 드러내기

### 2. 솔루션 설계

정의한 문제가 설계안에 잘 반영되어 있는지 본다.

중요한 점:

- 명확한 가설과 의도
- 사용자가 쉽게 이해할 수 있는 구조
- 사용자가 목적을 달성할 수 있는 UX
- 조건을 입력/해석/비교/확정하는 흐름의 자연스러움

우리 작업에서 보여줘야 할 것:

- 후보 시간이 왜 추천되었는지 설명 가능해야 함
- 필수 참석자와 선택 참석자의 차이가 UI에 반영되어야 함
- 사용자의 결정 부담을 줄여야 함
- 회의 시간 확정까지 막히지 않아야 함

### 3. 시각적 완성도

사용성뿐 아니라 심미적, 조형적 완성도를 본다.

중요한 점:

- 익숙한 UI pattern을 맥락에 맞게 다듬어 쓰기
- 정보 위계가 명확한 화면
- 시각적 detail로 완성도 높이기
- 단순히 예쁜 화면이 아니라 판단과 행동을 돕는 visual design

우리 작업에서 보여줘야 할 것:

- 시간 후보, 참석 가능 여부, 충돌, 추천 이유가 한눈에 읽혀야 함
- primary action과 secondary action 구분
- 색상/배지/spacing이 의미 있게 쓰여야 함
- prototype이 실제 product처럼 조작 가능해야 함

## 토스 Product Designer 관점

토스 Product Designer는 특정 도구나 화면 제작자로만 정의되지 않는다.

강조점:

- 어떤 화면을 그리는지가 아니라 **사용자 문제를 어떻게 정의하는지**
- 그 문제를 **어떻게 해결하는지**
- UX 설계, UI design, prototype까지 유저에게 가치를 전달하는 전 과정
- 정성/정량 feedback을 수집하고 니즈가 충족될 때까지 반복 개선
- 제품 경험에 대한 책임과 권한

이번 챌린지에서도 단순 산출물보다 다음 흐름이 중요하다.

```text
문제 발견 → 문제 정의 → 해결 가설 → 핵심 flow 설계 → prototype → 왜 이 방식인지 설명
```

- 추천된 후보 시간이 신뢰 가능하다는 것을 UI가 어떻게 설명할 것인가?
- 확정 전후로 참여자에게 어떤 feedback이 필요한가?

- 모든 기능을 넣으려다 핵심 flow가 흐려지는 것

### 집중할 핵심 flow 후보

```text
회의 만들기

→ 시스템이 후보 시간 추천
→ 추천 이유와 trade-off 확인
→ 최종 시간 확정
→ 공유/알림
```

## 제출 답변 최종안

### Q1. 사용자의 어떤 문제를 발견했나요? 왜 문제인가요?

벤치마킹한 서비스 중 Google Calendar와 Outlook은 참석자와 회의실의 가능·불가능 시간을 겹쳐 빈 시간을 찾을 수 있었고, Cal.com은 외부 사용자나 팀과의 예약에 초점을 두고 있었습니다. 다만 제가 검토한 핵심 흐름에서는 가능한 시간을 찾은 뒤 선택 참석자, 비선호 시간, 회의실 같은 조건을 하나씩 조정했을 때 어떤 대안이 생기고 누구에게 영향이 가는지 비교하기 어려웠습니다.

실제 일정 조율에서는 모든 조건을 만족하는 시간이 항상 최선은 아닙니다. 회의 목적에 따라 선택 참석자는 빠져도 됩니다. 비선호 시간이나 업무 시간, 회의실 조건도 주최자가 감수할 수 있습니다. 이런 차이를 확인하기 어렵다면 참석자에게 다시 묻거나 여러 일정을 직접 대조해야 합니다.

제가 발견한 문제는 추천 정확도가 아니라, 주최자가 조건의 중요도를 따져 현실적인 대안을 찾기 어렵다는 점이었습니다.

### Q2. 그 문제를 어떻게 해결했나요?

주최자가 조건의 우선순위를 바꿀 때마다 후보 시간과 선택의 영향이 바로 보이게 했습니다. 전체 참석자·필수 참석자, 업무 시간, 비선호 시간, 회의실 사용 가능 여부를 필터로 꺼내 추천 조건을 시스템 안에 숨기지 않았습니다. 참석자 역할, 등록된 일정과 선호, 회의실 예약처럼 시스템이 확인할 수 있고 주최자도 이해할 수 있는 조건만 사용했습니다.

필터를 바꾸면 후보 목록과 캘린더의 가능한 시간대가 함께 달라져 조건과 결과의 관계를 바로 알 수 있습니다. 후보는 기준 시각 이후 가까운 순으로 정렬하고 오늘로부터 며칠 뒤인지 표시했습니다.

후보를 선택하면 해당 날짜의 주간 캘린더로 이동해 기존 일정과 비교할 수 있습니다. 시간대에 마우스를 올리면 충돌 여부만 먼저 보여주고, 클릭하면 결정 영역에서 참석이 어려운 사람의 필수·선택 여부와 각자 어려운 이유를 확인할 수 있습니다. 추천을 자동 확정하지 않고 주최자가 직접 회의 시간을 선택하도록 했습니다.

### Q3. 왜 이 방식으로 설계했나요?

처음에는 캘린더 안에 추천 이유와 참석 조건을 모두 보여주려 했습니다. 하지만 월간 화면은 범위가 넓어 후보를 비교하기 어렵고, 일간 화면은 다른 날짜의 대안을 보기 어려웠습니다. 주간 화면은 한 주 흐름을 보기 좋았지만 이벤트 영역이 좁았습니다.

팝업이나 단계별 화면도 검토했습니다. 화면을 열고 닫으면 기존 일정의 맥락이 끊기고 후보를 연속해서 비교하기 어렵다고 보았습니다. 그래서 탐색 조건은 왼쪽 영역(Overview), 기존 일정은 가운데 캘린더(Context), 판단 근거는 오른쪽 영역(Decision)에 배치했습니다.

시스템은 가능한 시간을 계산할 수 있지만 어떤 참석자와 조건이 중요한지, 무엇을 포기할 수 있는지는 알지 못합니다. 조건과 영향을 드러내고 최종 선택을 주최자에게 맡긴 이유입니다. 참석 가능 인원순 정렬도 고민했지만 ‘전체·필수만’ 필터에 따라 순위가 달라져 가까운 시간순을 택했습니다.

세 영역을 한 번에 보여주면서 시선 이동이 많아졌고 오른쪽 수정이 왼쪽 후보와 중앙 캘린더를 바꾸는 역방향 흐름도 생겼습니다. 이번 과제에서는 이 한계를 감수하더라도 후보와 일정을 한 화면에서 이어서 비교하고, 정보를 늘 같은 자리에서 찾는 쪽을 택했습니다. 모바일에서 세 영역을 1–2–3으로 쌓는 것만으로는 같은 비교 경험을 유지하기가 어려워, 작은 화면에는 별도 IA가 필요하다고 판단해 이번 범위는 PC로 좁혔습니다.
